# Requirements: GymLog

**Defined:** 2026-01-27
**Core Value:** Track workout performance with proper data engineering â€” both usable as a personal training tool and impressive as a senior Data Engineer portfolio piece.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Exercise Management

- [x] **EXER-01**: User can create exercise with name, primary muscle group, and global/gym-specific flag
- [x] **EXER-02**: User can edit existing exercise details
- [x] **EXER-03**: User can delete exercise (with confirmation if used in templates)
- [x] **EXER-04**: User can view list of all exercises filtered by muscle group

### Gym Management

- [x] **GYM-01**: User can create gym with name and optional location
- [x] **GYM-02**: User can edit gym details
- [x] **GYM-03**: User can delete gym (with confirmation if has workout history)
- [x] **GYM-04**: User can view list of all gyms

### Workout Templates

- [x] **TMPL-01**: User can create workout template with name (e.g., "Upper A")
- [x] **TMPL-02**: User can add exercises to template in specific order
- [x] **TMPL-03**: User can set target rep range per exercise in template (e.g., 8-12)
- [x] **TMPL-04**: User can set optional replacement exercise per exercise in template
- [x] **TMPL-05**: User can reorder exercises within template
- [x] **TMPL-06**: User can edit template (add/remove/reorder exercises)
- [x] **TMPL-07**: User can delete template

### Workout Logging

- [x] **LOG-01**: User can start workout by selecting gym and template
- [x] **LOG-02**: User sees exercises in template order with target rep range and last session history
- [x] **LOG-03**: User can log each set with weight (kg) and reps
- [x] **LOG-04**: User can optionally add RIR to each set
- [x] **LOG-05**: User can switch from default exercise to predefined replacement
- [x] **LOG-06**: User can add custom one-off exercise (not saved to template)
- [x] **LOG-07**: User can start/stop rest timer between sets (configurable duration)
- [x] **LOG-08**: User can complete workout and save all logged data
- [x] **LOG-09**: User can cancel workout in progress (with confirmation)

### History & Analytics

- [x] **HIST-01**: User can view exercise history showing last 2 weeks of sets
- [x] **HIST-02**: History respects global vs gym-specific rule (gym-specific shows only current gym data)
- [x] **HIST-03**: User sees automatic PR detection when new PR is set
- [x] **HIST-04**: User can view estimated 1RM for exercises (Epley formula)
- [x] **HIST-05**: User can view list of all PRs per exercise

### Data Engineering

- [x] **DATA-01**: Data stored in star schema (fact_sets, dim_exercise, dim_gym, dim_workout_template)
- [x] **DATA-02**: Storage uses DuckDB-WASM with Parquet files in OPFS
- [x] **DATA-03**: dbt models transform raw events to staging to marts (compiled at build time)
- [x] **DATA-04**: Event sourcing: all changes stored as immutable events, views derived from events
- [x] **DATA-05**: dbt tests validate data quality (unique, not_null, relationships, custom rules)
- [x] **DATA-06**: dbt documentation generates data dictionary and lineage graphs
- [x] **DATA-07**: All raw events have audit columns (_created_at, _event_id)
- [x] **DATA-08**: Event processing is idempotent (replay-safe with natural keys)
- [x] **DATA-09**: Parquet files partitioned by month
- [x] **DATA-10**: Metrics layer defines 1RM, volume, PR calculations as single source of truth
- [x] **DATA-11**: Anomaly detection flags unusual values (e.g., weight jumped 50%+)

### Data Durability

- [x] **DURA-01**: User can export all data as Parquet file download
- [x] **DURA-02**: User can import data from Parquet backup file
- [x] **DURA-03**: App shows backup reminder after N workouts since last export

## Out of Scope

| Feature | Reason |
|---------|--------|
| Cloud sync/backup | Keeping local-first for v1, avoids infrastructure complexity |
| Multi-user support | Personal use only |
| Mobile native app | PWA covers mobile use case adequately |
| Social features | Not relevant for personal tracker |
| Calorie/nutrition tracking | Different domain, anti-feature per research |
| Gamification (streaks, badges) | Anti-feature per research, causes guilt |
| Unit conversion (lbs) | kg only for simplicity |
| OAuth/accounts | No backend, local data only |
